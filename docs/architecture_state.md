## 🎯 Purpose

This file is the **narrative memory of what has been built** — why modules exist, architectural decisions, module-to-business mapping. It is the human-curated complement to the **mechanical inventory** in [`architecture_state.auto.md`](./architecture_state.auto.md), which is auto-generated from the codebase at sprint closing and lists every table, route, Server Action, integration, and UI component.

**Split rule:**
- **Facts extractable from code** (table names, route paths, action names, migration filenames) → `architecture_state.auto.md` (script writes, no human edits).
- **Interpretation** (what a module does, why it was built this way, how modules relate) → this file.

**Writer:** Tech Lead appends at the end of every sprint (Workflow A, Step 7 Closing). Running the inventory script is also part of closing.
**Readers:** Tech Lead, `@spec-writer`, and every agent during initial context load — both files together.

**This file does NOT contain:**
- Target tech stack → see [`stack.md`](./stack.md)
- Repository layout → derivable from `ls` (framework structure lives in code, not in docs)
- Design system overview → see [`design_system/README.md`](../design_system/README.md)
- Memory layer rules → see [`agents/workflows/memory-layers.md`](../agents/workflows/memory-layers.md)
- Schema details → see [`schema_snapshot.json`](./schema_snapshot.json)
- Surprising traps → see [`APRENDIZADOS.md`](./APRENDIZADOS.md)
- Implementation details already visible in code
